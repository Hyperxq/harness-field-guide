Field guide · For Claude Code users

You don't just use Claude Code.
You harness it.

Out of the box, Claude Code is a very capable generalist that forgets everything between sessions and does whatever each prompt asks. A harness is the layer you build around it — rules, skills, sub-agents, hooks, and persistent memory — that turns "a chat that writes code" into a disciplined engineering system. This page maps one real, production harness so you can see what each piece does and why it exists.

01 · What problem does a harness solve?

The model is smart. The process is what's missing.

Every failure mode you've hit with an AI agent — code that looks right but isn't, scope that silently grows, context lost after a long session, "done" that doesn't actually solve the problem — is a process failure, not an intelligence failure. A harness doesn't make the model smarter; it makes the workflow around the model impossible to skip.

🔧 Vanilla Claude Code

  • One conversation does everything: plans, codes, reviews itself
  • Memory ends when the session ends
  • Quality depends on how good each prompt is
  • Nothing stops it from touching auth code or skipping tests
  • "Done" means "the model says it's done"

⚙️ Harnessed Claude Code

  • Roles are split: planners plan, executors build, evaluators judge
  • Decisions and lessons persist across sessions in a memory backend
  • Quality lives in reusable skills, not in each prompt
  • Hooks deterministically block risky writes and bypasses
  • "Done" means independent verification passed, with evidence

The core idea: the human directs, the AI executes — but you can only direct what you can constrain. The harness is how you encode your engineering judgment once, so you don't have to re-prompt it every day.

03 · Memory & sight

Two MCP servers the whole harness leans on

An agent has two structural blindnesses: it forgets — everything dies with the context window — and it can't see a codebase cheaply — understanding structure through grep costs dozens of calls and a bloated context. The harness plugs in one MCP server for each, and almost every layer above depends on them.

🧠 engram — the diary

  • Solves: amnesia between sessions and after context compaction
  • Stores decisions, specs, lessons, and pipeline state under stable topic keys (sdd/{change}/spec…)
  • Sub-agents are born empty: they receive topic references and read only what they need
  • /resume works because state lives outside the conversation
  • If lost, it's unrecoverable — the "why" behind decisions isn't in the code

🕸️ codegraph — the map

  • Solves: expensive, blind code exploration
  • Live-indexed graph of every symbol, call, and dependency in the workspace
  • "How does X work?" is one call returning the relevant source — not thirty greps
  • Callers and impact analysis follow real symbols, including dynamic dispatch that text search can't see
  • If lost, rebuild it from the repo in seconds — it's derived, never authored

Rule of thumb: codegraph answers "how is the code connected today?" — engram answers "what did we decide, and why?". The planner explores with codegraph; every phase persists to engram. One is a map you can redraw; the other is a diary you can't rewrite.

04 · The operating model

Planner, Executor, Evaluator — never the same mind

The harness splits every change into three roles. The Planner defines what to build. The Executor builds it. The Evaluator judges the result against the spec — and crucially, evaluators are launched blind: they see the artifacts, never the builder's reasoning, so they can't inherit its blind spots. Executor and Evaluator form a loop (build → verify → fix → verify) capped at 3 iterations before a human is pulled in.

05 · The pipeline

SDD: spec-driven development, phase by phase

Three commands drive everything: /plan runs the Planner until a signed, verified plan exists; /build runs the Executor–Evaluator loop; /evaluate runs the final judgment before archiving. Every phase writes its artifact to memory before the next one starts.

The techniques inside /plan

The phases aren't improvised — each one applies a named engineering technique:

TechniqueWhereWhat it buys you
🪜 Abstraction ladderexplore → designPlanning starts at the altitude the problem demands — user flow → system → architecture → code — and descends. Big changes never start at code; that's how you get clever solutions to the wrong problem
🧾 Given/When/Then + stable REQ-IDsspecEvery requirement is a testable scenario with an ID that never changes, so slices, tests, and verification all trace back to the same contract
📜 ADRsdesignEach substantive decision is recorded with its why and alternatives; the best get promoted to project level on archive, so future changes inherit the reasoning
🍰 SPIDR vertical slicingsliceSlices cut through all layers (UI to data) and each delivers something demoable — never "backend first, frontend someday"
🚶 Walking skeletonslice → applySlice S-000 is the thinnest end-to-end path, built first: it proves the architecture actually connects before any feature hangs off it
🧪 Test derivation + strict TDDdesign → applyThe design maps every REQ-ID to the tests that will prove it; in strict TDD mode the executor must write those tests before the code
🙈 Anti-anchoringevery reviewJudges and council personas receive only the artifacts under review — never the author's reasoning. An evaluator that reads your logic inherits your blind spots
06 · Triage

Not every change deserves the full ceremony

The first phase sizes the change and routes it. A typo fix should never pay the cost of a council review — and a payments change should never escape it. One override beats size: anything touching a sensitive area (auth, payments, privacy, data migration) is forced onto the heavyweight path no matter how small it looks.

ClassExamplePathCouncil personas
XSTypo, config tweakInline — the coordinator just does itNone
SSmall isolated fixInline, with a quick verifyNone
MNew endpoint, refactorFull pipeline, light councilSituational (QA, architect…)
LFeature across layersFull pipeline + every gateFull: architect, BA, PM, QA + conditionals
XLEpic-sized initiativeBecomes a program: decomposed into L/M sub-changes with explicit contracts between them, integration-verified after each oneFull, per sub-change

🏛️ The council — many lenses, one plan

For M and L changes, the planning phases don't rely on a single reviewer. The coordinator convenes a council: persona sub-agents, each examining the same artifact through one professional lens. They run in parallel and blind — no persona sees another's opinion, only the artifact under review — and the coordinator synthesizes their perspectives afterwards. It is not design-by-committee: personas advise, the pipeline decides.

PersonaTheir lensWhen they join
🏗️ ArchitectLayering, dependency direction, abstraction boundaries, ADRsAlways on L; situational on M
📊 Business analystUser value, acceptance criteria, scope clarity, scenarios from the user's sideAlways on L — explore, propose, spec
📈 Product managerPrioritization, value vs. cost, followups tracked as debt instead of vanishingAlways on L
🧪 QA engineerAdversarial test thinking: edge cases, failure modes, mutation-resistant assertionsAlways on L — spec, design, verify
🔐 Security engineerAttack surface, validation, threat modeling, secrets handlingWhenever the change touches a sensitive area
🗄️ DBASchema design, indexes, migration safety, backfillsWhen the data layer is materially affected
🎨 UX designerUser flows, accessibility, mental models, error statesWhen the change has a user-facing surface
✍️ Tech writerNaming, contract documentation, error-message languageWhen public APIs or external contracts change

Why it works: a single reviewer optimizes for one thing and goes blind to the rest. Parallel, independent lenses surface the tradeoffs before they're welded into code — and XS/S changes skip the council entirely, because that's triage doing its job: ceremony proportional to risk.

07 · Anatomy of one turn

What actually happens when you type /plan

The coordinator never does the heavy lifting itself — it resolves context, launches a sub-agent with exactly the context it needs, then persists the result. That's the whole trick: a thin conversation thread that survives, and disposable workers that don't need to.

  autonumber
  participant U as 👤 You
  participant O as 🎛️ Coordinator, the main session
  participant H as 🚧 Hooks
  participant A as 🤖 Sub-agent, fresh context
  participant M as 🧠 Memory backend
  U->>O: /plan add-csv-export
  O->>M: read project context + past lessons
  M-->>O: stack, sensitive areas, conventions
  O->>H: launch triage sub-agent
  H-->>O: allowed, prerequisites exist

Step 9 is why a crash, a context compaction, or tomorrow's fresh session can pick up exactly where this one stopped: state lives outside the conversation.

08 · The gates

Trust the model, verify the work

Gates are checkpoints that can halt or reroute the pipeline. Each asks a different question — and the reviewers are always independent agents that never saw the author's reasoning.

GateWhenThe question it asksIf it fails
🚪 Plan verifyAfter slicing, before buildingIs this plan sufficient? Two blind judges: one checks problem fit, one simulates being the executorRoutes back to the exact phase that has the gap
🧪 Verify in-loopAfter each slice batchDoes the code match the spec, with real test evidence?Fix attempt; 3 strikes → human
🧹 SimplifyStart of /evaluateIs there duplication, dead weight, or over-engineering in the full diff?Applies safe cleanups; non-blocking
🏁 Verify finalBefore archiveDoes the whole change satisfy every requirement end to end?Re-spec or re-design, by failure category
🧭 StewardPost-design and pre-archiveAre we solving the original problem — or just shipping code that passes tests?Halts with an outcome gap; escalates "is it usable?" to the human
🏛️ Architecture auditBefore archive (M/L)Did the change violate the architecture baseline?Blocks archive until reconciled — or overridden with a persisted reason
⚔️ Adversarial reviewLarge or sensitive changesTwo independent judges attack the diff — blind, spec-onlyFix iterations; unresolved findings need a human decision

Four of these deserve a closer look, because they carry the philosophy of the whole system.

📡 The sensors — verify in-loop

During /build, every slice batch passes through a fast, delta-focused verification: run the real tests, hold the slice against its acceptance criteria and REQ-IDs, report with evidence. Think of them as sensors on a production line, not a final inspection at the exit. A failure gives the executor a fix attempt; three failures escalate to a human — the loop is never allowed to spin forever. The payoff: drift gets caught while it is one slice old and cheap to fix, instead of being discovered at the end, fossilized under ten more slices built on top of it.

🙈 The blind judges

A blind judge is an independent sub-agent that receives only the artifacts under review — the spec, the diff, the acceptance criteria — never the author's reasoning, never the coordinator's conversation, never the other judge's opinion. In plan-verify, one judge checks problem fit ("does this plan solve the stated problem?") while the other simulates being the executor ("could I build this with zero questions?"). In adversarial review, two judges attack the final diff armed with the spec alone. The blindness is the point: an evaluator that reads your logic inherits your blind spots and ends up agreeing with you. Two independent judges with different lenses catch failures that two copies of the same reviewer never would.

🧭 The voice of conscience — the steward

The steward is born during /plan: right after design, it writes the north-star memo — the outcome this change is chasing, as opposed to the outputs it will produce ("users stop losing drafts", not "an autosave endpoint exists"). At the very end, before archive, it returns for the reckoning: did we deliver on the original problem? Is it usable? Is it significant — or is it just code that passes tests? It answers what it can analyze, and for the questions only a human can judge, it escalates instead of faking a verdict. Why it exists: verification proves the code matches the spec — the conscience proves the spec still matched the problem. On long changes, those two drift apart in silence.

🧹 Why simplify runs before the final verdict

Order is everything here, for two reasons. First, the final verification must judge the code that will actually ship — cleaning up after the verdict would invalidate the verdict. Second, simplify is allowed to be aggressive precisely because the final verify runs after it as a safety net: if a cleanup accidentally changes behavior, the gate catches it. And the two never mix concerns — simplify hunts quality (duplication, dead weight, over-engineering, code at the wrong altitude across the full diff, not slice by slice), while verify hunts correctness. A reviewer asked to do both does both badly.

Why so many? Each gate exists because a specific failure once slipped through without it. Verification tells you the code matches the spec; the steward tells you the spec matched the problem. Different failures, different gates.

10 · Build your own

You don't need all of this on day one

This harness grew gate by gate, each piece added after a real failure. If you're starting yours, grow it in the same order the pain arrives:

Start withPain it removesClaude Code mechanism
1 · A constitutionRe-explaining your standards in every sessionCLAUDE.md
2 · Persistent memoryDecisions and lessons evaporating between sessionsMCP memory server, or the built-in memory directory
3 · SkillsGood workflows living only in your headskills/*/SKILL.md
4 · Separate reviewersThe author grading its own homeworkSub-agents, launched blind
5 · HooksRules the model can forget under pressurePreToolUse / Stop hooks in settings.json
6 · A pipelineBig changes with no auditable planOrchestration rules + persisted phase artifacts

The takeaway: a harness is your engineering culture, written down where the agent can't ignore it. The model brings the intelligence; you bring the judgment — and the harness is how your judgment keeps working while you sleep.
