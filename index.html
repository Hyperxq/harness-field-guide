Field guide · For Claude Code users

You don't just use Claude Code.
You harness it.

Out of the box, Claude Code is a very capable generalist that forgets everything between sessions and does whatever each prompt asks. A harness is the layer you build around it — rules, skills, sub-agents, hooks, and persistent memory — that turns "a chat that writes code" into a disciplined engineering system. This page maps one real, production harness: what each piece does, why it exists, and the software-craftsmanship ideas it's all built on. And one thing before you dive in — this harness is alive: what you're reading is a snapshot of a system that grew failure by failure and will keep changing. Read it as a base to adapt, not a recipe to copy.

01 · What problem does a harness solve?

The model is smart. The process is what's missing.

Every failure mode you've hit with an AI agent — code that looks right but isn't, scope that silently grows, context lost after a long session, "done" that doesn't actually solve the problem — is a process failure, not an intelligence failure. A harness doesn't make the model smarter; it makes the workflow around the model impossible to skip.

🔧 Vanilla Claude Code

  • One conversation does everything: plans, codes, reviews itself
  • Memory ends when the session ends
  • Quality depends on how good each prompt is
  • Nothing stops it from touching auth code or skipping tests
  • "Done" means "the model says it's done"

⚙️ Harnessed Claude Code

  • Roles are split: planners plan, executors build, evaluators judge
  • Decisions and lessons persist across sessions in a memory backend
  • Quality lives in reusable skills, not in each prompt
  • Hooks deterministically block risky writes and bypasses
  • "Done" means independent verification passed, with evidence

The core idea: the human directs, the AI executes — but you can only direct what you can constrain. The harness is how you encode your engineering judgment once, so you don't have to re-prompt it every day.

02 · The craftsmanship underneath

The harness invents nothing — it enforces what masters already knew

Every mechanism in this system is a classic software-craftsmanship practice, automated and made impossible to skip. This is the real curriculum: if you understand these ten ideas, you understand the harness — and you're a better engineer even with the AI switched off.

PracticeThe ideaWhere the harness enforces it
🎯 Problem-first thinkingNever touch code until you can say who is hurting, what pain, and why nowTriage refuses to start without a problem statement and explicit scope
🧾 Specification before codeRequirements as testable Given/When/Then scenarios with IDs that never changeThe spec phase; every slice, test, and verdict traces back to a REQ-ID
🪜 Right altitudeThink user flow → system → architecture → code, descending — never start at codeExplore and design enter the abstraction ladder at the height triage demands
🚶 Walking skeletonBuild the thinnest end-to-end path first, to prove the architecture connectsSlice S-000, always built before any feature
🍰 Vertical slicingEvery increment cuts through all layers and is demoable — no "backend first, UI someday"SPIDR slicing in the slice phase
🧪 Test-driven developmentTests derive from the spec and come before the codeDesign maps each REQ-ID to its tests; strict TDD mode in apply
📜 Decisions on the recordSignificant choices written down with their why and the alternatives (ADRs)The design phase; the best ADRs get promoted to project level on archive
🙈 Independent reviewThe author never grades their own homework — and reviewers stay blind to the author's reasoningThe council, plan verify, adversarial review — every evaluator in the system
⚖️ Ceremony proportional to riskA typo fix and a payments change do not deserve the same processTriage's XS→XL routing, plus the sensitive-area override
📚 Continuous learningEvery finished change should teach the next oneArchive extracts lessons-learned into permanent memory

This is why "AI is a tool" isn't a slogan here. The human's judgment lives inside these practices — the harness just makes them non-negotiable. Learn them and any agent becomes more powerful in your hands; skip them and no amount of tooling saves you.

03 · The building blocks

Six layers, each answering one question

Everything in this harness is one of six kinds of thing. Claude Code provides the mechanisms; the harness is what you build with them.

Artifact storepaper trail
"Where does the plan live?" Specs, designs, slices, and verification reports are persisted artifacts — in memory topics, as files (openspec/changes/…), or synced to Jira, Confluence, or GitHub. The plan is a document you can audit, not a vibe in the chat.
openspec/changes/{change}/

And the harness polices itself. The hook layer doesn't only watch the agent — it points inward too. A validator script enforces a size budget on the constitution (new subsystems must enter as small pointers to contract files, never as resident bloat), checks that mirrored tables across files stay in sync, and pipeline-guard refuses to launch any phase whose prerequisites are missing from persisted state. The harness treats its own configuration as code under review — that's what keeps a living system from decaying into a junk drawer.

04 · Memory & sight

Two MCP servers the whole harness leans on

An agent has two structural blindnesses: it forgets — everything dies with the context window — and it can't see a codebase cheaply — understanding structure through grep costs dozens of calls and a bloated context. The harness plugs in one MCP server for each, and almost every layer above depends on them.

🧠 engram — the diary

  • Solves: amnesia between sessions and after context compaction
  • Stores decisions, specs, lessons, and pipeline state under stable topic keys (sdd/{change}/spec…)
  • Sub-agents are born empty: they receive topic references and read only what they need
  • /resume works because state lives outside the conversation
  • If lost, it's unrecoverable — the "why" behind decisions isn't in the code

🕸️ codegraph — the map

  • Solves: expensive, blind code exploration
  • Live-indexed graph of every symbol, call, and dependency in the workspace
  • "How does X work?" is one call returning the relevant source — not thirty greps
  • Callers and impact analysis follow real symbols, including dynamic dispatch that text search can't see
  • If lost, rebuild it from the repo in seconds — it's derived, never authored

Rule of thumb: codegraph answers "how is the code connected today?" — engram answers "what did we decide, and why?". The planner explores with codegraph; every phase persists to engram. One is a map you can redraw; the other is a diary you can't rewrite.

05 · The operating model

Planner, Executor, Evaluator — never the same mind

The harness splits every change into three roles. The Planner defines what to build. The Executor builds it. The Evaluator judges the result against the spec — and crucially, evaluators are launched blind: they see the artifacts, never the builder's reasoning, so they can't inherit its blind spots. Executor and Evaluator form a loop (build → verify → fix → verify) capped at 3 iterations before a human is pulled in.

🗺️ The living map — watching it all happen

Every phase you just saw also feeds a living visual map. After each phase completes, a visualization skill re-renders one stable-URL page per change: the problem and scope, every requirement with progress dots (spec'd → implemented → verified), the architecture diagrams, the slices, and the build board. It's the harness's observability layer — you don't ask "where are we?", you open the map. It's resilient by design: if session state is ever lost, a rebuild mode reconstructs the entire map from memory, spec files, tickets, and git. And it's non-blocking — a broken map never halts the pipeline. The page you're reading right now grew out of that same skill's design system, which is the kind of dogfooding a living harness should be doing.

07 · Triage

Not every change deserves the full ceremony

The first phase sizes the change and routes it. A typo fix should never pay the cost of a council review — and a payments change should never escape it. One override beats size: anything touching a sensitive area (auth, payments, privacy, data migration) is forced onto the heavyweight path no matter how small it looks.

ClassExamplePathCouncil personas
XSTypo, config tweakInline — the coordinator just does itNone
SSmall isolated fixInline, with a quick verifyNone
MNew endpoint, refactorFull pipeline, light councilSituational (QA, architect…)
LFeature across layersFull pipeline + every gateFull: architect, BA, PM, QA + conditionals
XLEpic-sized initiativeBecomes a program: decomposed into L/M sub-changes with explicit contracts between them, integration-verified after each oneFull, per sub-change

🏛️ The council — many lenses, one plan

For M and L changes, the planning phases don't rely on a single reviewer. The coordinator convenes a council: persona sub-agents, each examining the same artifact through one professional lens. They run in parallel and blind — no persona sees another's opinion, only the artifact under review — and the coordinator synthesizes their perspectives afterwards. It is not design-by-committee: personas advise, the pipeline decides.

PersonaTheir lensWhen they join
🏗️ ArchitectLayering, dependency direction, abstraction boundaries, ADRsAlways on L; situational on M
📊 Business analystUser value, acceptance criteria, scope clarity, scenarios from the user's sideAlways on L — explore, propose, spec
📈 Product managerPrioritization, value vs. cost, followups tracked as debt instead of vanishingAlways on L
🧪 QA engineerAdversarial test thinking: edge cases, failure modes, mutation-resistant assertionsAlways on L — spec, design, verify
🔐 Security engineerAttack surface, validation, threat modeling, secrets handlingWhenever the change touches a sensitive area
🗄️ DBASchema design, indexes, migration safety, backfillsWhen the data layer is materially affected
🎨 UX designerUser flows, accessibility, mental models, error statesWhen the change has a user-facing surface
✍️ Tech writerNaming, contract documentation, error-message languageWhen public APIs or external contracts change

Why it works: a single reviewer optimizes for one thing and goes blind to the rest. Parallel, independent lenses surface the tradeoffs before they're welded into code — and XS/S changes skip the council entirely, because that's triage doing its job: ceremony proportional to risk.

09 · Verification

Three checkpoints, because "wrong" has three shapes

Most workflows verify once, at the end. This harness verifies three times, at three altitudes — and it's not paranoia, it's the same logic as the test pyramid: unit, integration, and end-to-end tests all exist because each catches a failure the others structurally cannot see. Same here. A plan can be insufficient. A slice can drift from its spec. A finished change can pass every test and still fail the problem. Different failure, different moment, different reviewer.

1 · 🚪 Verify the plan — before a single line of code

After slicing, the plan faces two blind judges, each with a different responsibility. Judge A checks problem fit: does this plan actually solve the stated problem, inside the stated scope? Judge B simulates being the executor: "if I had to build from these artifacts alone — no conversation, no shared context — could I? What's missing, ambiguous, undecided?" That no-context simulation isn't theater: sub-agents really are born without memory, so a plan that only works with the planner's context in your head is a plan that will fail in execution. Alongside the judges, the steward's foresight — the voice of conscience — writes the north-star memo: the outcome this change is chasing ("users stop losing drafts", not "an autosave endpoint exists"), so there is something honest to reckon against at the end. Gaps route back to the exact phase that owns them, max 3 rounds. A gap found here costs minutes.

2 · 🧪 Verify in the loop — while building

The sensors. After every slice batch: run the real tests, hold the slice against its acceptance criteria and REQ-IDs, report with evidence. Fast and delta-focused — a sensor on the production line, not an inspection at the exit. A failure gives the executor one fix attempt; three failures escalate to a human, so the loop can never spin forever. The payoff: drift gets caught while it is one slice old and cheap to fix, instead of being discovered at the end, fossilized under ten more slices built on top of it.

3 · 🏁 Verify the whole — before archive

Three movements, in strict order. First, a simplify pass cleans the full diff — duplication, dead weight, over-engineering. It runs before the verdict so the verdict judges the code that will actually ship, and it can afford to be aggressive precisely because final verification comes right after it as a safety net. Then verify final checks every requirement end-to-end — and when the change is large or touches a sensitive area, it summons two adversarial judges with different responsibilities, blind, armed with the spec and the diff alone, each attacking from their own lens. Finally the circle closes: the architecture audit asks whether the change violated the baseline, and the steward returns for the reckoning, holding the result against its own north-star memo from checkpoint 1 — we said we were chasing this outcome: did we deliver it? Is it usable? Is it significant — or just code that passes tests? What only a human can judge, it escalates instead of faking a verdict.

CheckpointWhat only it can catchWho judgesCost of catching it here
🚪 PlanA plan that can't be executed without context — or solves the wrong problemTwo blind judges + steward foresightMinutes — nothing is built yet
🧪 In-loopA slice that drifted from its specThe sensor, with real test evidenceOne slice — small blast radius
🏁 FinalA whole that doesn't hold together — or passes every test without solving the problemFinal verifier + two adversarial judges + arch audit + steward reckoningHours — but the alternative is production

Why so many, and why all different? Because redundancy checks the same thing twice, while layering checks different things once — and only layering scales. The reviewers differ (every judge is blind: artifacts only, never the author's reasoning, never each other's opinions — an evaluator that reads your logic inherits your blind spots). The questions differ: sufficiency, then fidelity, then wholeness. And the timing puts each check as close as possible to the mistake it can catch, where fixing is cheapest. Remove any one layer and a specific class of failure ships in silence.

11 · Build your own

You don't need all of this on day one

This harness grew gate by gate, each piece added after a real failure. If you're starting yours, grow it in the same order the pain arrives:

Start withPain it removesClaude Code mechanism
1 · A constitutionRe-explaining your standards in every sessionCLAUDE.md
2 · Persistent memoryDecisions and lessons evaporating between sessionsMCP memory server, or the built-in memory directory
3 · SkillsGood workflows living only in your headskills/*/SKILL.md
4 · Separate reviewersThe author grading its own homeworkSub-agents, launched blind
5 · HooksRules the model can forget under pressurePreToolUse / Stop hooks in settings.json
6 · A pipelineBig changes with no auditable planOrchestration rules + persisted phase artifacts

The takeaway: this harness is alive — and yours should be too. Everything on this page is a snapshot of a system that grew failure by failure and will keep evolving as the failures change. So don't copy it — fork it. Use it as a base: keep what solves problems you actually have, delete what doesn't, and let your own pain add the next gate. A harness is your engineering culture written down where the agent can't ignore it, and culture, by definition, is never finished.
